# CodeDB Yazelix Nushell Runtime Bridge

Generated: 2026-07-02
Task: CDB049

## Boundary

CodeDB should treat Yazelix as the runtime packager and session launcher, not as the source-of-truth store. The CodeDB store is the higher-fidelity source data plane: it owns file-to-table capture, blob semantics, Rust crate/source extraction, cargo/build evidence, and structured query layers. Envctl remains the environment/export layer that can serialize runtime state and later materialize files from CodeDB-backed tables when needed.

## Yazelix Runtime Surfaces

Yazelix packages Nushell as a first-class runtime tool:

- `packaging/runtime_tool_registry.nix` declares `nushell` with package `pkgs.nushell` and command `nu`
- `packaging/mk_runtime_tree.nix` links the tracked `nushell/` tree into the runtime and makes the generated `yzx` frontdoor prepend packaged `nu` to `PATH`
- `shells/posix/runtime_env.sh` exports `YAZELIX_NU_BIN` to `$runtime_dir/libexec/nu` when the bundled runtime Nu exists, then falls back to host `nu`
- `shells/posix/yazelix_nu.sh` fails fast if `YAZELIX_NU_BIN` is unavailable, writes a launch-local generated config under `$YAZELIX_STATE_DIR/generated/nushell/config.nu`, and executes `nu --login --env-config /dev/null --config <generated-config>`

That means CodeDB tests and integration should prefer the Yazelix-provided `YAZELIX_NU_BIN` when proving behavior inside a Yazelix runtime, while still keeping host-`nu` compatibility as an explicit matrix row.

## Managed Config Shape

The tracked `nushell/config/config.nu` is intentionally small. It:

- exits unless `IN_YAZELIX_SHELL` or `YAZELIX_RUNTIME_DIR` is present
- imports the Nushell standard library modules
- disables the welcome banner
- sources `~/.local/share/yazelix/initializers/nushell/yazelix_init.nu`
- clears the right prompt for managed panes
- keeps only a tiny alias/function surface
- sources `~/.local/share/yazelix/initializers/nushell/yazelix_extern.nu`

Do not wire CodeDB by patching this tracked config directly. A durable integration belongs in package/runtime metadata plus generated initializer or extern state.

## Generated Initializer Ownership

Yazelix generates shell initializers under `~/.local/share/yazelix/initializers/`. For Nu, `rust_core/yazelix_core/src/initializer_commands.rs` targets `initializers/nushell`, writes tool-specific `*_init.nu` files, and writes the aggregate `yazelix_init.nu`.

Home Manager activation calls `yzx_control generate_shell_initializers` after runtime materialization. The docs explicitly say generated initializers are regenerated by Yazelix and should not be edited manually.

CodeDB should therefore use generated state in one of these forms:

- generated initializer or extern wiring produced by a Yazelix-controlled command
- a documented user registry mode for users who opt into persistent Nu plugin registration
- transient test mode using `nu --plugins '[.../nu_plugin_codedb]'` with a temporary HOME

## Plugin Registration Modes

Use three distinct modes so tests and runtime behavior do not blur ownership:

1. Transient proof mode
   - Launch with `nu --plugins '[.../nu_plugin_codedb]'`
   - Set a temporary HOME and XDG roots
   - Do not mutate the user plugin registry

2. User registry mode
   - User intentionally runs `plugin add <path-to-nu_plugin_codedb>`
   - User imports with `plugin use codedb`
   - This mode is not used by automated package gates unless HOME is temporary

3. Yazelix generated bridge mode
   - Yazelix runtime package exposes `codedb` and `nu_plugin_codedb`
   - Generated initializer or extern state advertises the plugin path and commands
   - Future package variables may include `YAZELIX_CODEDB_BIN` and `YAZELIX_CODEDB_PLUGIN_BIN`

## CodeDB Versus Envctl

The CodeDB store should be more accurate than envctl because it records source artifacts as typed rows and blob-backed layers instead of only exporting environment state. Nu's native file-to-table strengths are useful at the interactive edge, but the Rust plugin must keep the durable semantics:

- every scanned file has stable identity, content hash, path metadata, and blob handling
- Rust crates, manifests, source items, macro/proc-macro facts, include/build-script evidence, and native-library hints remain typed tables
- secrets and unsafe captures are explicitly classified rather than leaked through display or stderr
- envctl exports can be imported as one source table, then converted back to files or environment materialization only when requested

Envctl is the converter and deployment bridge. CodeDB is the high-fidelity codedb store and query surface.
